#!/usr/bin/env bash
set -euo pipefail          # exit on error · unset vars · pipe failure

# Load GCP credentials and project config (gitignored)
source .env

# Incremental-load guard — requires explicit written confirmation
if [[ "${1:-}" == "--incremental" ]]; then
  read -p "Run incremental load? Type YES to confirm: " CONFIRM
  [[ "$CONFIRM" == "YES" ]] || exit 1
fi

Execution model

Steps run sequentially. Any non-zero exit immediately aborts the entire pipeline via set -euo pipefail. No partial state left behind.

Data flow: GCS → local → BigQuery

CSVs live in a GCS bucket. Before running locally, download them: gsutil -m cp gs://$GCS_BUCKET/data/* data/. Steps 3–4 ingest to olist_raw; step 5 builds the star schema in olist_analytics; step 6 writes dbt staging views + mart tables to olist_analytics_staging / olist_analytics_marts.

Secrets

GCP_PROJECT_ID, GOOGLE_APPLICATION_CREDENTIALS, GCS_BUCKET_NAME loaded from .env — gitignored, never committed.

--incremental flag

Activates APPEND mode for core ingestion and SHA-256 hash detection for geolocation. Requires the literal string YES to proceed.

Asset Groups

  • olist_ingest_only — DQ + ingestion groups only

Meltano ELT layer

The ingestion group assets are named meltano_* to represent the Singer/Meltano ELT design intent (tap-csv → target-bigquery). Due to google-cloud-bigquery 3.x API incompatibility with available target-bigquery variants, the assets execute Python ingestion scripts directly — preserving the conceptual ELT separation while remaining production-stable.

BigQuery as materialisation target

Each asset group writes to a dedicated BigQuery dataset: ingestion → olist_raw · warehouse → olist_analytics (star schema: 7 dims + 4 facts) · dbt staging → olist_analytics_staging (9 views) · dbt marts → olist_analytics_marts (7 tables). Destination dataset is recorded in each materialisation metadata.

02 · Orchestration
Asset Dependency Graph
Two parallel ingestion branches converge before dbt transformation

Parallel execution

star_schema_build and dbt_staging both depend on the two ingestion assets — Dagster's planner runs them concurrently, saving several minutes per run.

Incremental geo

SHA-256 hash of the source CSV is checked every run. If unchanged, the load is skipped and Dagster records skipped: true in materialisation metadata.

Failure propagation

Asset failures propagate as skipped (not failed) to downstream assets — preserving the distinction between a self-failure and a blocked execution.

Jobs, Schedule & Resources
Three purpose-built jobs let operators run targeted subsets of the pipeline

Job Definitions

  • olist_daily_pipeline — all 10 assets; used by schedule and GHA
  • olist_ingest_only — DQ + ingestion; fast reload without dbt
  • olist_dbt_only — warehouse + dbt + charts; model-only reruns

Schedule

  • Name: olist_daily_at_0600_utc
  • Cron: 0 6 * * * (06:00 UTC)
  • Status: STOPPED — GitHub Actions is primary
  • Enable via Dagster UI as GHA fallback

DbtCliResource

  • Injected into all 4 dbt assets at job startup
  • Project dir: dbt_project/
  • Streams dbt events to the Dagster event log
  • Raises DagsterDbtCliRuntimeError on failure

Upgrade Path → @dbt_assets

  • Replace 4 dbt assets with a single @dbt_assets
  • Requires pre-compiled manifest.json
  • Exposes per-model lineage in asset catalog
  • Needs custom DagsterDbtTranslator

The 5-Job Pipeline Chain
Sequential dependencies via needs:, with conditional execution on the ingest job
1
ingest
Downloads CSVs from GCS → DQ validation (57+ core + 30 geo) → ingests 10 core tables and geolocation to BigQuery olist_raw.
if: schedule == '0 18 * * *' || (workflow_dispatch && full_pipeline == true)
2
dbt-staging
Runs dbt deps && dbt run --select tag:staging — 9 staging models (8 views + 1 table for geolocation). Runs even when ingest is skipped.

3
dbt-marts
Executes star schema DDL (warehouse/schema.sql), then dbt run --select tag:mart — 7 mart tables with up to 4 parallel threads.
if: always() && needs.dbt-staging.result == 'success'
4
integration-tests
Runs dbt test — 181+ tests (staging schema + mart schema + 6 SQL singular assertions) — then generates 8 analytics charts.
if: always() && needs.dbt-marts.result == 'success'
5
notify
Always runs. Writes a structured summary to $GITHUB_STEP_SUMMARY and posts a Slack alert (success or failure) via webhook curl.
if: always()   ·   needs: [ingest, dbt-staging, dbt-marts, integration-tests]
03 · CI/CD
Conditional Logic & Notifications
The always() pattern ensures notify runs even when ingest is intentionally skipped
# Job 1: ingest — cron/dispatch conditional
jobs:
  ingest:
    if: |
      github.event.schedule == '0 18 * * *'
      || (github.event_name ==
          'workflow_dispatch'
          && github.event.inputs.full_pipeline
             == 'true')
    steps:
      - name: Download CSVs from GCS
        run: |
          gsutil -m cp \
            gs://$GCS_BUCKET/data/* data/
      - name: DQ + Ingest
        run: |
          python data_quality/dq_validation.py
          python data_quality/dq_geolocation.py
          python ingestion/ingest_to_bigquery.py
          python ingestion/ingest_geolocation.py
# Job 2: skips gracefully if ingest was skipped
  dbt-staging:
    needs: [ingest]
    if: |
      always()
      && needs.ingest.result != 'failure'
      && needs.ingest.result != 'cancelled'
# Job 5: notify — always() pattern
  notify:
    needs: [ingest, dbt-staging,
            dbt-marts, integration-tests]
    if: always()
    steps:
      - name: GitHub Step Summary
        run: |
          echo "## Pipeline Results" \
            >> $GITHUB_STEP_SUMMARY
          echo "| Job | Status |" \
            >> $GITHUB_STEP_SUMMARY

      - name: Slack — success
        if: |
          needs.integration-tests.result
          == 'success'
        run: |
          curl -X POST $SLACK_WEBHOOK_URL \
            -d '{"text":"✅ succeeded"}'

      - name: Slack — failure
        if: |
          needs.ingest.result == 'failure'
          || needs.dbt-staging.result
             == 'failure'
          || needs.dbt-marts.result
             == 'failure'
          || needs.integration-tests.result
             == 'failure'
        run: |
          curl -X POST $SLACK_WEBHOOK_URL \
            -d '{"text":"❌ failed"}'

Why always() matters

Without it, a skipped upstream job propagates a "skipped" status that blocks notify. always() breaks that propagation — notify fires regardless of whether ingest ran, was skipped, failed, or succeeded.

Summary
Three Layers, One Pipeline
Each orchestration layer serves a distinct purpose — complementary, not redundant

run_pipeline.sh

  • Purpose — Developer tool
  • Trigger — Manual, ad-hoc
  • Environment — Local server
  • Steps — 7 sequential
  • Fail mode — set -euo pipefail
  • Best for — Debug, smoke test, first-time setup

Dagster + Meltano

  • Purpose — Observability & retry
  • ELT layer — Meltano (Singer) → Python scripts
  • Trigger — UI / schedule (STOPPED)
  • Environment — Local Dagster server
  • Assets — 10 (parallel branches)
  • Fail mode — Failure + skip downstream
  • Best for — Materialisation audit trail

GitHub Actions

  • Purpose — Production orchestration
  • Trigger — 2 crons + workflow_dispatch
  • Environment — ubuntu-latest runner
  • Jobs — 5 sequential
  • Fail mode — always() notify
  • Best for — Automated daily production runs

Production flow (02:00 SGT)

GitHub Actions downloads CSVs from GCS → runs 57+ DQ checks → ingests 10 tables + geolocation to BigQuery → runs 9 dbt staging models → runs 7 dbt mart tables + star schema DDL → runs 181+ tests → generates 8 analytics charts → notifies Slack. Total run time: ~15–25 minutes.

Dagster schedule is STOPPED intentionally

Enabling it while GHA is active causes double-runs. To migrate away from GHA, flip DefaultScheduleStatus.STOPPED → RUNNING in orchestration/dagster/__init__.py.

SHA-256 incremental guard

In both ingest_geolocation.py and the Dagster geo asset. The 1M-row CSV rarely changes — skipping the load on an unchanged file saves ~3 minutes per run.
